# 错误桶分析

逐题对比 4 种格式的回答，按 5 类错误分桶。

## 错误分类定义

| 桶 | 定义 |
|---|------|
| **边界污染** | 噪声/注释/追加文本被当成有效配置 |
| **优先级丢失** | 知道有规则，但冲突时不会正确裁决 |
| **规则漏读** | 某条规则直接没看到或没提取出来 |
| **规则混淆** | 把 soft 当 hard，或弄混规则适用条件 |
| **脑补补全** | 生成不存在的规则/例外/解释 |

## 核心发现

### 1. XML 的真正优势：格式语法提供了天然的噪声防火墙
XML 在 I3 中表现突出 — 模型能够识别出 `# OVERRIDE` 不是有效的 XML 注释语法（XML 用 `<!-- -->`），因此拒绝将其视为配置。
**这不是偶然的，这是格式固有的结构化优势。**

### 2. DSL 的致命弱点：没有格式级别的噪声防护
DSL 使用 `#` 作为可能的注释标记，所以 `# OVERRIDE:` 看起来像一条合法的指令。
DSL 的规则语义虽然清晰，但它缺乏一个"什么是有效语法"的刚性边界。

### 3. JSON5 的意外优势：同样有语法防火墙
JSON 的严格语法也能拒绝注入 — `# OVERRIDE` 不是合法 JSON。

### 4. Markdown 是最脆弱的：没有语法概念
Markdown 没有"有效/无效语法"的概念，任何文本都是合法的 Markdown。
这意味着任何追加的噪声都可能被当作配置。

### 5. 规则混淆 (Q8) 是格式无关的
所有格式在 Q8 上都犯了同样的错误 — 混淆了有条件允许和无条件允许。
这说明这个问题来自 LLM 自身的推理，不是格式导致的。

## GPT 猜测验证

GPT 预测：
> XML 错在"看到了结构，但没抓住行为语义"
> DSL 错在"规则读懂了，但没有外层边界保护，容易被干扰文本污染"

**验证结果：GPT 猜对了。**

- XML 唯一的弱项是 B5（优先级裁决失败）— 看到了结构但没正确执行行为冲突
- DSL 在 I1/I3 被污染 — 规则读得准，但没有语法级的防护墙

## 结论：单格式增强方向

### XML 增强方向：补语义显式性
- 在 conflict_resolution 中使用更显式的属性（如 `winner="R3"` 而不是自然语言描述）
- 预计可修复 B5 类错误

### DSL 增强方向：补硬边界隔离
- 添加固定的区域分隔符（如 `<<<CONFIG_BEGIN>>>` / `<<<CONFIG_END>>>`）
- 或使用带校验的区块标记
- 预计可修复 I1/I3 类边界污染错误

### 这两个增强是否可以在单一格式内完成？
- **XML 补语义**：可以，用更丰富的属性和子标签
- **DSL 补边界**：可以，加固定分隔符

**因此，先做单格式增强是正确的路线。暂不需要混合方案。**
