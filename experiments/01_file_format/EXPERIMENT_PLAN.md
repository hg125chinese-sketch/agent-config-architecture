# 实验 01: 哪种表示法最像给 LLM 写的可遵循宪法

## 唯一研究问题

哪种配置表示方式最能最大化 LLM 对 agent 规则的理解与遵循准确度？

## 核心指标: LLM Fidelity Score

| 子项 | 权重 | 测法 |
|------|------|------|
| 约束遵循率 | 40% | 20 条 hard constraints，实际遵守几条 |
| 冲突裁决正确率 | 25% | 10 个冲突场景，是否按配置优先级处理 |
| 关键信息抽取准确率 | 20% | 字段和规则提取是否完整准确 |
| 幻觉惩罚 | 15% | 每脑补一条不存在的规则，扣分 |

辅助记录（不参与排名，仅用于诊断）：
- Token 数：准确度相同时的决胜项
- 解析合法率：上场门槛

## 候选格式（4 种）

| 编号 | 格式 | 押注理由 |
|------|------|----------|
| A | Markdown + YAML frontmatter | 基线，行业现状 |
| B | XML | 标签边界显式，层级清晰，LLM 训练语料丰富 |
| C | JSON5 | key-value 明确，歧义低，抽取友好 |
| D | 自定义 DSL | 语义标签高度显式，理论上限最高 |

## 测试材料（2 类）

1. **Skill 文件** — 直接决定行为
2. **Identity 文件** — 最容易"看了但没照做"

## 测试任务（4 种）

### Task 1: 规则抽取
给配置文件，问具体问题：触发条件、hard constraints、优先级、禁止项。
测基础理解。

### Task 2: 冲突场景执行
给会触发规则冲突的用户输入，看模型按哪条执行。
测优先级裁决。

### Task 3: 边界条件执行
部分满足 trigger / 多规则同时命中 / 有例外条款。
测精确判断。

### Task 4: 抗干扰执行
配置文件后追加噪声文本，看准确度是否下降。
测格式的抗上下文污染能力。

## 执行流程

1. 编写 2 份基线素材（Skill + Identity），语义固定
2. 将同一语义转写为 4 种格式，严格等价
3. 为每种任务设计测试用例（附标准答案）
4. 逐格式跑测试，记录结果
5. 算分，排名，分析典型失败案例
